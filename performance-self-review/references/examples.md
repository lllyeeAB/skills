# 示例

这些示例用于学习“如何转换”，不要机械照抄句子。

## 示例 1：从工作记录到绩效段落

### 原始内容的典型模式

- Linux 构建与 Docker 运行适配
- Local API / CDP 能力支持
- 多线程下载与断点续传问题修复
- RPA 框架 block、翻译器和拟人化能力建设
- SEO 或内部业务支撑类任务

### 更好的输出方式

本考核周期内，我围绕团队在客户端工程化、自动化能力建设和客户需求交付上的重点方向持续推进工作。核心投入一方面放在 RPA 框架能力完善，补齐了流程执行、输入拟人化、结果标准化和翻译器衔接等底层能力；另一方面持续推进 Linux/Docker 场景下的运行与交付能力建设，并完善 Local API/CDP、下载链路和稳定性相关能力，支撑客户接入和业务使用。整体上，我承担的工作既包含底层通用能力建设，也包含面向具体场景的需求交付，对团队交付效率、系统稳定性和后续复用都有较直接的支撑作用。

### 为什么这样写更好

- 先归纳主线，而不是照着时间线重复
- 技术工作和团队价值之间有明确连接
- 语气克制，适合直接提交

## 示例 2：长度约束很强时的处理

### 用户要求

- 共三个维度
- 每个维度控制在 300-400 字
- 语气务实，可直接提交
- 不要出现“根据日报”这类说法

### 正确处理方式

- 每个维度只写一个完整段落
- 每段只挑最能代表该维度的工作
- 删除暴露生成过程的措辞
- 用事实支撑，语气保持克制

## 示例 3：应该避免的写法

### 较差的输出

我根据本周期的日报进行了总结，充分体现了本人积极主动、认真负责的工作态度，完成了大量任务并为团队创造了重要价值。

### 问题

- 暴露了生成过程
- 用空泛评价替代事实
- 没有真正对应考核维度
- 语气很模板化，说服力弱
