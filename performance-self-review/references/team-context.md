# 团队背景

这个文件只用于约束语气、重点和默认理解方式，不能当成补事实的依据。

## 默认绩效口径

- 务实、克制、可直接提交
- 聚焦真实工作和具体贡献
- 避免堆砌大词和过度包装
- 语气更像工程师向直属经理总结工作，而不是宣传文案

## 团队里通常算“有效内容”的东西

- 在某条工作主线上有持续承担，而不是零散罗列任务
- 对团队目标有直接支撑，例如能力建设、交付效率、稳定性、客户支持、业务推进
- 能体现完整闭环，例如方案、开发、联调、排障、验证、发布、跟进
- 能处理跨模块、跨平台或复杂边界问题
- 面对阻塞问题时能快速响应并推动收敛

## 这个团队常见的工作类型

这些类型经常出现在原始材料里，适合被归纳为主要贡献主线：

- App 工程化与平台适配
- Linux / Docker 运行与交付支持
- Local API、CDP 或客户接入能力
- RPA 框架或自动化能力建设
- 性能、下载链路和稳定性优化
- SEO、内部工具或业务支撑系统
- electron故障排查、性能优化
- UI细节打磨、疑难杂症攻克
- 服务器搭建经验沉淀、坑点

## 各维度的推荐写法

### 1. 组织目标贡献 / 客户需求导向

重点突出：

- 与团队重点目标的对齐
- 可复用能力建设
- 对客户场景的直接支撑
- 对业务或运营需求的支持
- 对效率、稳定性、可交付性的价值

避免：

- 只列技术动作，不解释价值
- 只说“做了很多”，但不说明对谁有用

### 2. 任务需求按时交付

重点突出：

- 多项任务并行推进的节奏感
- 任务拆解和闭环推进能力
- 在开发、联调、发布阶段及时处理问题
- 在遇到阻塞时仍然保障整体交付节奏

避免：

- 在原材料没有支撑时强说“零延期”
- 过程写得过细，像周报复读

### 3. 任务需求难度

重点突出：

- 跨端、跨模块、跨平台的复杂度
- 架构、抽象、框架、基础设施类工作
- 权限、并发、端口、运行时、平台差异等复杂边界
- 既需要方案设计，也需要工程落地的任务

避免：

- 把普通功能开发夸成高难任务
- 只写“难度较高”，不解释难在哪里

## 需要避免的表达

- 根据日报 / 结合日报 / 从以上工作可以看出
- 充分体现了 / 全方位 / 高效赋能 / 显著领先
- 积极主动 / 认真负责 / 沟通顺畅，如果没有具体事实支撑
- 纯按日期展开的流水账

## 缺少明确指标时的安全表达

没有精确数据时，优先使用：

- 提升了可交付性
- 增强了稳定性
- 提高了接入灵活性
- 支撑了业务推进
- 沉淀了可复用能力
- 降低了后续协作和维护成本
